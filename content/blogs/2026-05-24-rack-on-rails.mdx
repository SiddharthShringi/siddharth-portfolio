In this article, we are going to understand how Rack fits into an HTTP request
response cycle, why it was introduced, and what problem it solves.
What are Rack middlewares and their use cases?
This article will also explain in brief about web-servers, web-frameworks,
how they talk to each other, and in the end, we are going to build our mini framework that can handle http request and send http response.

A web framework needs to communicate with the web server in order to get the request from the client and send a response to the client.

In the context of Ruby, there are different web servers ( like Puma, Unicorn, or Thin) and web frameworks(Like Rails, Sinatra, Hanami).

# 1. The Problem

Rack solves a specific issue in the HTTP request-response lifecycle. To understand Rack, we need to go back in the world where Rack doesn't exist.

Before Rack was introduced, web servers and web frameworks were often incompatible or tightly coupled.

It means that if we are building a web server, we have to manually write code to interface with every application or framework it might support, such as Rails, Sinatra, or even a plain Ruby script.

On the other side, if we are building a framework, we had to write custom handlers for every web server we wanted to support.

This scenario created a complexity in both side, where creating a new server or a framework will require major integration work. This introduced a lot of code duplication, because frameworks essentially are performing same tasks.

![Before Rack Example](/Images/blog/2026-05-24-rack-on-rails/before-rack.webp)

We can see in the image 1 the complexity of web servers working with web frameworks.

# 2. Understanding Rack

To solve this complexity and issues, there was a need for a solution that could untangle these complexities.

That's where Rack comes into the picture. According to documentation definition - **Rack** is a minimal, modular, and adaptable **interface** designed to unify how Ruby web servers, web frameworks, and middleware communication.

Basically, Rack is a protocol that defines a specific architecture that any Ruby object or class must follow to be `Rack-compliant`.

## The Rack Protocol

### For Rack Application

Any framework or application has to respond to the call method in order to be Rack Compliant.

Let's understand this.

It means to be Rack compliant, the Ruby object should respond to the call method.

- A call method takes one argument - `env.`
- An `env` is a hash that contains environment data about the incoming request and server.
- And the `call` method returns an array of three specific values
  1.  Status Code **(Integer) **e.g. 200, 402
  2.  Headers **(Hash)**
  3.  Response Body **(Enumerable)**

A very simple example of Rack compliant object can be like this:

```ruby
class MyRackApp
  def call(env)
    [200, { 'content-type' => 'text/plain' }, ['Hello, World!']]
  end
end
```

### For Rack Server

The necessary condition for web-server to be Rack compliant is that it must proactively look for a Ruby object( Web Application or Framework) and **invoke its** **call** **method**.

To do that Rack-Compliant web server must have a `run` method which takes a Rack application as an argument and calls its `call` method when a request is received.

![After Rack Example](/Images/blog/2026-05-24-rack-on-rails/after-rack.webp)

This protocol solves code duplication of complicated connections for every web sever and web framework.

# 3. The Rack gem and Middleware

We've understood the Rack protocol - It is a set of rules that allows a web server and web application to communicate with each other.

The Rack gem is a Ruby library that provides the infrastructure needed to build an application that follows the Rack protocol. It ships with a collection of build-in **middlewares**.

## What are middlewares?

A Rack middleware is a component that sits between the web server and the web application.

It catches the request before it reaches the application, and/or the response before it is sent back to the client.

It is a Rack-compliant object, which means it can respond to the `call` method, but instead of returning the final response, it calls the next `call` method of middleware in the chain.

There are some functionalities that can be implemented in each web application or framework like logging, authentication, and session management.

Without middlewares, the application itself has to handle them. This led to problems like code duplication.

With middlewares, we can implement these functionalities in a modular way and reuse them across different applications and frameworks.

![After Rack Example](/Images/blog/2026-05-24-rack-on-rails/middleware.webp)

Here are some of the middlewares provided by the Rack gem:

- **Rack::Files:** Another component used for serving static files.
- **Rack::Session:** A component used for managing user sessions.
- **Rack::Logger:** A component used for logging request and response data.
- **Rack::MethodOverride:** Supports PUT and DELETE HTTP methods by allowing the method to be overridden if the params[:_method] is set

Every Rack middleware is a Rack application, but not every Rack application is a Rack middleware.
So structurally, a middleware is itself a valid Rack application. The difference is intent and behaviour - a middleware holds a reference to the next layer and calls it, whereas a plain Rack application does not.

You can see the entire list of middlewares provided by Rack gem [here](https://github.com/rack/rack/tree/main/lib/rack)

## Middleware Stack

A middleware stack is the ordered chain of middleware objects. Requests travel down the stack toward the application, and responses travel back up the stack toward the client.

It allows each middleware to inspect or modify both the request and response.

## Config.ru and Rack DSL

The `config.ru` is a configuration file that specify the instructions for rack application.
This is the file that tells Rack:

- Which application should handle the request?
- Which middleware should be placed around that application?

We can use a different file name if we want to, but by convention everyone uses `config.ru`

```ruby
# config.ru

use RequestLogger
use Authentication

run HelloWorldApp.new
```

In this example, we see two keywords, `use` and `run`. These are part of Rack DSL (Domain Specific Language).

**use:** Adds middleware to the stack.
**run:** Specifies the final Rack application.

## Rackup Command

The generic Rack application is started with the `rackup` command.
Previously, it was part of the `Rack` gem, but as of Rack 3, the `rackup` command has been moved to a separate `rackup` gem.

```bash
~ rackup
```

This command does the following things:

- Reads `config.ru` file.
- Builds a middleware stack.
- Creates the Rack Application
- Start a web server(Thin, Unicorn, Puma etc...)

## Custom Middleware

We can also create our own custom middleware by creating a class that responds to the `call` method. Let's create a simple middleware that logs the request path and method. We will use this middleware in our mini web application later.

```ruby
class RequestLogger
  def initialize(app)
    @app = app
  end

  def call(env)

    request = Rack::Request.new(env)
    puts "Received #{request.request_method} request for #{request.path}"
    @app.call(env) # Call the next middleware or application
  end
end
```

Let's understand what this middleware is doing:

First, we defined the custom middleware class `RequestLogger`, which will log the information about incoming requests.

Next, we initialize the app. This line means we are wrapping the next middleware and passing the data env, so that it can be part of the middleware stack.

The call(env) method is part of the Rack protocol for Rack applications and middleware. In this method, we first create a `Rack::Request` object.
So that instead of manually using `env["REQUEST_METHOD"]` and `env["PATH_INFO"]` we can use a helper object.

Then we are logging it
`puts "Received #{request.request_method} request for #{request.path}"`

And finally, it is passing the request to the next middleware or application.

We can modify the response inside the middleware. See example below.

```ruby
def call(env)

 status, headers, body = @app.call(env)

 headers["X-Powered-By"] = "My Middleware"

 [status, headers, body]
end
```

Now every response will include the header `X-Powered-By`.

# 4. Building a Mini Web Application with Rack

Now that we have understood the basics of Rack and middlewares, let's build a mini web application using Rack.

## Step 1: Create a project

Let's create a new directory for our Rack application and install the Rack gem.

```bash
~ mkdir my_rack_app
~ cd my_rack_app
~ bundle init
~ echo "gem 'rack'" >> Gemfile
~ echo "gem 'rackup'" >> Gemfile
~ echo "gem 'puma" >> Gemfile
~ bundle install
```

Here we have created a new directory, initialized a new bundle, and added `rack` and `rackup` gems to our Gemfile.

## Step 2: Create a Rack Application

Let's create a simple Rack application that responds with "Hello, World!, I'm a Rack application."
For that, we will create a new file `application.rb` and add the following code:

```ruby

class MyRackApp
  def call(env)
    [
      200,
      { "Content-Type" => "text/plain" },
      ["Hello, World!, I'm a Rack application."]
    ]
  end
end
```

Here, we added a `call` method that returns a status code of 200, a content-type header of text/plain, and a response body according to Rack protocol.

## Step 3: Create a config.ru file

Now we need to create a `config.ru` file that will specify how to run our application and which middlewares to use.

```ruby
# config.ru

require_relative 'application'
run MyRackApp.new
```

## Step 4: Start the Rack application

To start the application, we can use the `rackup` command in the terminal.

## Step 5: Routing

Let's add some routing to our application. We will modify our `MyRackApp` class to handle different paths.

```ruby
class MyRackApp
  def call(env)
    request = Rack::Request.new(env)

    case request.path
    when "/"
      response("Home")
    when "/about"
      response("About Page")
    else
      response("Not Found", 404)
    end
  end

  private

  def response(body, status = 200)
    [
      status,
      { "content-type" => "text/plain" },
      [body]
    ]
  end
end
```

In this code, we are using `Rack::Request` to parse the incoming request and determine the path. Based on the path, we are returning different responses.

## Step 6: Using Custom Middleware

Remember, we created a custom middleware `RequestLogger` earlier. We will slightly modify it.
Let's create a new file `request_logger.rb` and add the following code:

```ruby
class RequestLogger
  def initialize(app)
    @app = app
  end

  def call(env)
    request = Rack::Request.new(env)

    puts "#{request.request_method} #{request.path}"

    status, headers, body = @app.call(env)

    headers["x-app"] = "Rack Demo"

    [status, headers, body]
  end
end
```

This middleware will log the request method and path, and also add a custom header `x-app` to the response.

We will also add another Rack middleware, Rack::ContentLength, which automatically adds the `Content-Length` header to the response.

Now let's use them in our application. We will modify our `config.ru` file to include the middleware.

```ruby
# config.ru

require_relative 'application'
require_relative 'request_logger'

use Rack::ContentLength
use RequestLogger
run MyRackApp.new
```

Let's restart our application and see the changes using curl.

