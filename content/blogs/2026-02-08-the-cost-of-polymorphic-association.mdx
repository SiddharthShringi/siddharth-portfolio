Polymorphism refers to the ability of something to take multiple forms.
In Rails, a polymorphic association allows one model to belong to different
types of models through a single association.

I've read about polymorphic associations many times and understood what
they are in theory. But when deciding whether to use them,
I lacked clear criteria for making that choice.

In this article, we are going to explore the reasoning behind
polymorphic associations and the trade-offs they introduce.

# The Problem

Let's think of a scenario where we are building a learning platform where:

- A quiz can have comments.
- A lesson can have comments.
- A post can have comments.

What are the ways we can implement these relationships?

**First approach:** We will create separate comment tables for each model.

```plaintext
Quiz -> QuizComments(id, text)
Lesson -> LessonComments(id, text)
Post -> PostComments(id, text)
```

All three tables store identical data.
The only difference here is the parent model.
It's a schema duplication and unnecessary separation
of identical behaviour.

Let's consider alternative ways to implement these
associations.

**Second approach:** Create a single table which belongs
to all three parents. To implement this relationship, we will create
a comment table where `quiz_id`, `lesson_id`, and `post_id`
will be foreign keys.

| id  | text                | quiz_id | lesson_id | post_id |
| --- | ------------------- | ------- | --------- | ------- |
| 1   | "comment on quiz"   | 1       | NULL      | NULL    |
| 2   | "comment on lesson" | NULL    | 1         | NULL    |
| 3   | "comment on post"   | NULL    | NULL      | 1       |

While this approach avoids duplication,
it introduces other issues. As the number of parent models grows,
the table expands horizontally, resulting in many NULL values.
The database becomes messy and doesn't scale well.

So how can we model this relationship without duplicating
tables or making the schema messy?. This is where a
polymorphic association becomes useful.

# Polymorphic Association

![Rails Initialization Process](/Images/blog/2026-02-08-the-cost-of-polymorphic-association/polymorphic-association.webp)

The core idea behind a polymorphic association is simple: instead of storing multiple foreign keys,
we store two columns - `commentable_type` and `commentable_id`
Together, these columns identify both:

- the type of parent model.
- And the specific record it belongs to.

This denotes that comment belongs to something polymorphic (_commentable:_ something that can take multiple forms).

| id  | text                | commentable_type | commentable_id |
| --- | ------------------- | ---------------- | -------------- |
| 1   | "comment on quiz"   | Quiz             | 1              |
| 2   | "comment on lesson" | Lesson           | 1              |
| 3   | "comment on post"   | Post             | 1              |

Each row stores the parent model's class name and the corresponding record id.
This allows a single comment table to associate with multiple models.

For example first row says that the comment with id 1 belongs to Quiz with id 1.
Second row says the comment id 2 belongs to Lesson with id 1 and so on.

A `polymorphic association` is useful when child model behaves the same regardless
of parent model type, but needs to belong to different type of models.

# Modeling the Relationship

```ruby
#Comment model

class Comment < ApplicationRecord
  belongs_to :commentable, polymorphic: true
end

# Quiz model

class Quiz < ApplicationRecord
  has_many :comments, as: :commentable
end


# Lesson model

class Lesson < ApplicationRecord
  has_many :comments, as: :commentable
end

# Post model

class Post < ApplicationRecord
  has_many :comments, as: :commentable
end
```

The as: :commentable option must match the association name defined in belongs_to.

# Migrations

```ruby
create_table :comments do |t|
  t.text :body
  t.references :commentable, polymorphic: true, null: false
  t.timestamps
end
```

Here Rails generates `commentable_type` and `commentable_id` columns in comments table.

Let's see how Rails uses these columns when creating records.

```ruby
quiz.comments.create(text: "This is a great quiz")
lesson.comments.create(text: "This is a great lesson")
post.comments.create(text: "This is great post")
```

In the above code Rails create a following record.

| id  | text                   | commentable_type | commentable_id |
| --- | ---------------------- | ---------------- | -------------- |
| 1   | This is a great quiz   | Quiz             | 1              |
| 2   | This is a great lesson | Lesson           | 1              |
| 3   | This is a great post   | Post             | 1              |

In a polymorphic association:

- The relationship is represented by two columns:
  `commentable_type` and `commentable_id`.
- `commentable_type` acts as a type discriminator,
  storing the name of the parent model (e.g., Quiz, Lesson, Post).
- `commentable_id` stores the id of the associated parent record.

Rails uses both values together to determine which model and which
specific record the comment belongs to.

Notice that this design does not use a traditional database-level foreign key.
Rails resolves the association at the application layer.

# The Cost

## No Database Level Foreign-Key Constraint

When we have normal foreign-key relationship

```sql
FOREIGN KEY (post_id) REFERENCES posts(id)
```

This allows database to enforce foreign key constraint, and this prevent invalid references, and handle cascading deletes safely.
Cascading deletes is a database feature that automatically deletes related records in child tables when a record in a parent table is deleted.

In a polymorphic association, this is not possible. Since `commentable_id` can refer to different tables depending on `commentable_type`, the database cannot enforce a traditional foreign key constraint.
Therefore, database-level cascading deletes won't work for polymorphic associations.

As a result:

- Referential integrity is not enforced at the database-level.
- Orphaned records can exist if parents are deleted outside Rails.
- We must cleanup on application-level logic (e.g using dependent: :destroy).

```ruby
class Post < ApplicationRecord
  has_many :comments, as: :commentable, dependent: :destroy
end
```

When you delete a post:
Rails finds all comments where `commentable_type = 'Post'` AND `commentable_id = [post.id]`,
Rails deletes each comment (running any callbacks they might have)
and then Rails deletes the post.

Polymorphic associations trade database-level guarantees for application-level flexibility.

> Referential integrity is the rule that guarantees those relationships stay correct. If one record points to another record, the database ensures that the referenced record actually exists.

## Query Complexity Increases

In a normal association, joining tables is straightforward:

```sql
JOIN posts ON posts.id = comments.post_id
```

in Polymorphic Association, the join must also filter by type.

```sql
JOIN posts
  ON posts.id = comments.commentable_id
  AND comments.commentable_type = 'Post'
```

Now we have to write additional conditional query for each parent and thant makes the query long and complex.
In small apps this is fine, but in a analytic heavy system, complex queries are harder to optimize.

## Limited Database Optimization

Rails automatically creates a composite index when we add this line in migration file.

```ruby
t.references :commentable, polymorphic: true, null: false
```

We only need to add the index if we're adding polymorphic columns to an existing table.

```plaintext
add_index :comments, [:commentable_type, :commentable_id]
```

The index allows queries like below to execute efficiently.

```sql
SELECT * FROM comments
WHERE commentable_type = 'Post'
AND commentable_id = 10
```

But despite that the database query optimizer can't use the index as effectively with polymorphic associations compared to normal foreign keys reason being no referential integrity at database level.

## Renaming is hard

In Polymorphic Association, the parent model class names are stored as a string in type column.
Suppose if we rename `Post` model to `Article`. then the database will still have comment records where:

```ruby
commentable_type = "Post"
```

You will have to migrate the data manually, and this introduces operational risk,
especially in production systems with large datasets.

# When Polymorphic Association Is a Good Choice

Polymorphic associations work well when:

- The child model behaves uniformly for all parent models.
- If we want extensibility without altering schema.
- Examples: attachments, likes, activity logs, comments.

# When to Be Cautious

- Strict database-level integrity is required.
- The child model does not behaves uniformly for all parent models.
- The system is analytic-heavy, It might require complex queries.
- Very large tables may lead to large composite indexes, which can impact performance.

Polymorphic associations are a powerful Rails feature that make it easy
to share one child model across many parent models, but there are also some costs.

They provide application-level flexibility, but they trade away database-level
integrity and structural explicitness. The right choice depends on what your system values more.
